# CLI Authentication Flow

## Overview

`swatcher login` authenticates the CLI using a browser-based flow — no manual API key copy-paste required. The user signs in through the UI, and the CLI automatically receives and stores a long-lived API key.

This is modelled after tools like `gh auth login` and `vercel login`.

---

## Flow

```
swatcher login
     │
     ▼
1. CLI generates a random `state` token (CSRF protection)
2. CLI starts a temporary HTTP server on localhost:9876
3. CLI opens the browser to:
   https://signal-watcher-ui.merendis.workers.dev/auth/cli
     ?state=<random>
     &redirect_uri=http://localhost:9876/callback
     │
     ▼
4. User signs in (or is already signed in) in the browser
5. UI validates the session, then generates a one-time `code`
   and stores { code, userId, expiresAt } server-side
6. UI redirects to:
   http://localhost:9876/callback?code=<one-time-code>&state=<state>
     │
     ▼
7. CLI receives the callback, verifies `state` matches
8. CLI calls POST /api/auth/cli/exchange  { code }
9. UI validates the code (unused, not expired), creates a
   Better Auth API key for the user, marks code as used
10. UI returns { apiKey, baseUrl }
11. CLI saves credentials to ~/.config/swatcher/credentials.json
12. Local HTTP server shuts down
13. CLI prints "Logged in. API key saved."
```

---

## Components

### `swatcher login` (CLI)

- Generates `state` (random UUID)
- Binds `localhost:9876` (or next free port)
- Opens browser via `open` / `xdg-open`
- Waits for the callback request (timeout: 5 minutes)
- Verifies `state`, calls `/api/auth/cli/exchange`
- Writes `~/.config/swatcher/credentials.json`:
  ```json
  {
    "apiKey": "sk_...",
    "baseUrl": "https://signal-watcher-ui.merendis.workers.dev"
  }
  ```
- All subsequent commands read from this file (overridden by `SIGNAL_WATCHER_URL` + `SIGNAL_WATCHER_TOKEN` env vars for scripting)

### `GET /auth/cli` (UI — new Astro page)

- Validates `state` and `redirect_uri` params (redirect_uri must be `http://localhost:*`)
- If the user has no session → shows the sign-in form, preserving params in a hidden field or query string through the sign-in redirect
- If the user has a session → immediately generates a one-time code, stores it, and redirects to `redirect_uri`
- One-time code: random UUID, expires in 10 minutes, stored in D1

### `GET /auth/cli/callback` (UI — new Astro page, optional)

- Not strictly needed — the redirect goes directly to `localhost`. Included here in case the UI needs an intermediate step (e.g. showing a "You can close this tab" message before redirecting).

### `POST /api/auth/cli/exchange` (UI — new API endpoint)

- Body: `{ code: string }`
- Looks up the code in D1, checks it is unused and not expired
- Calls Better Auth `api-key/create` on behalf of the user (server-side, using the userId from the stored code row)
- Marks the code as used
- Returns `{ apiKey: string, baseUrl: string }`
- Returns 400 if code is invalid/expired/already used

---

## Security considerations

- **`state` parameter** — random UUID generated by the CLI, verified on callback. Prevents CSRF / open redirect attacks.
- **`redirect_uri` allowlist** — UI only accepts `http://localhost:<port>` redirect URIs. Rejects anything else.
- **One-time code** — server-side, expires in 10 minutes, single-use. Even if intercepted in transit (unlikely on localhost), it cannot be reused.
- **API key scope** — the created API key belongs to the authenticated user. The CLI never sees the user's password or session cookie.
- **Local server** — binds only to `localhost`, not `0.0.0.0`. Not reachable from the network.

## Credentials file

```
~/.config/swatcher/credentials.json
```

```json
{
  "apiKey": "sk_live_...",
  "baseUrl": "https://signal-watcher-ui.merendis.workers.dev"
}
```

Environment variables (`SIGNAL_WATCHER_URL`, `SIGNAL_WATCHER_TOKEN`) take precedence over the credentials file, so automated scripts and CI pipelines are unaffected.

## Database changes

A new `cli_auth_code` table is added to the UI's D1 schema:

```ts
export const cliAuthCode = sqliteTable('cli_auth_code', {
  id:        text('id').primaryKey().$defaultFn(() => crypto.randomUUID()),
  code:      text('code').notNull().unique().$defaultFn(() => crypto.randomUUID()),
  userId:    text('user_id').notNull(),
  usedAt:    text('used_at'),
  expiresAt: text('expires_at').notNull(),
  createdAt: text('created_at').notNull().$defaultFn(() => new Date().toISOString()),
});
```

## AuthEntrypoint (backend)

Once the CLI has an API key, the backend Worker needs to validate it against Better Auth rather than a static `API_TOKEN`. This is done via a `WorkerEntrypoint` on the UI Worker:

```ts
// signal-watcher-ui: src/entrypoints/auth.ts
export class AuthEntrypoint extends WorkerEntrypoint<Env> {
  async validateToken(token: string): Promise<{ userId: string } | null> {
    const result = await auth.api.verifyApiKey({ body: { key: token } });
    if (!result?.key) return null;
    return { userId: result.key.referenceId };
  }
}
```

The backend Worker calls this via Service Binding instead of comparing against `API_TOKEN`:

```ts
// cloudflare-signal-watcher: auth middleware
const result = await c.env.AUTH.validateToken(token);
if (!result) return c.json({ error: 'Unauthorized' }, 401);
c.set('userId', result.userId);
```

This replaces the static `API_TOKEN` secret on the backend entirely.
